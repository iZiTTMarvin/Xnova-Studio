## Purpose

这是一份持续更新的工作设计文档。

用途：

- 固化当前已经确认的产品与架构决策
- 避免后续长对话中结论漂移或遗忘
- 作为后续 `/plan-eng-review`、设计稿、实现拆解的统一基线

## Product Goal

`Xnova Studio` 的 v1 目标不是先接入大量外部 Agent，而是先把 `Xnova` 自己打通成一个完整可用、可长期演进的桌面编码 Agent 产品。

目标用户：

- 你自己长期使用
- 身边不会熟练使用 `Claude Code`、`Codex` 等工具的同学
- 需要从 0 到 1 开发项目、或接手已有项目继续开发的用户

## Core Positioning

### Xnova

- `Xnova` 是主 Agent
- 不是外部 Agent 的壳
- 未来可以调度外部 Agent，但产品中心始终在 `Xnova`

### Xnova Studio

- `Xnova Studio` 是桌面宿主
- UI 风格接近 `Codex App`
- 但产品内核要更强 `project-aware`

### XForge

- `XForge` 是开发模式
- 不是主 Agent
- 不是 `agent profile`
- 它代表一套更强的 orchestration / workflow 策略

### XForge Positioning in v1

`XForge` 在 `Xnova Studio v1` 中的定位是：

- 已定义的产品概念
- 已预留的模式层扩展位
- 明确的后续演进方向

但不是当前 v1 的完整交付范围。

v1 中与 `XForge` 相关的已预留能力包括：

- 顶部 `Standard / XForge` 模式切换
- `allowed_modes`
- `recommended_mode`
- mode 与 agent/profile 分层

v1 明确不要求：

- 完整复刻或原生实现 `xdev + gstack + Trellis` 的全链路编排
- 完整端到端 workflow orchestration
- 默认强依赖并自动安装 `gstack`、`Trellis`

后续方向：

- `XForge = Xnova Studio 内置的原生端到端智能开发模式`
- 后续将吸收 `xdev` 的编排思想、`gstack` 的工作流能力、`Trellis` 的 spec/task 管理方式

## v1 Scope

v1 要同时支持以下主链路：

1. 新建项目，从 `0 → 1` 完成初始化、写代码、跑测试
2. 打开一个开发到一半的已有项目，直接继续开发
3. 在以上两种场景中自然使用 `subagent`

## Architecture Decisions

### Runtime Shape

- 架构采用 `shared runtime + dual host`
- CLI 和桌面端共享同一套核心运行时
- 不走“桌面壳里套一个真正终端 CLI”的粗糙路线

### v1 Priority

- 先把 `Xnova` 自身打通
- 先打通主 Agent、SubAgent、自定义 Agent、Project-aware 配置、Memory、MCP、会话体验
- 外部 Agent 放到后续阶段，不作为 v1 依赖

### Non-goals for v1

- 暂不优先做沙箱/审批模式
- 暂不优先做大量外部 Agent 接入

## UX Decisions

### UI Principle

- UI 外观接近 `Codex App`
- 表层体验是聊天驱动
- 底层数据模型是项目驱动

### Left Sidebar v1

左侧一级入口采用：

- 快速聊天
- 搜索
- Agents
- 项目
- 聊天
- 工具
- 设置

说明：

- `工具` 入口承载 `MCP / Skills`
- `XForge` 不作为左侧一级入口，而是顶部模式切换
- `Plugins` 暂不作为 v1 左侧一级入口，优先放入设置或工具相关二级界面
- `工具` 页在 v1 中偏状态页，而不是重型管理页
- `工具` 页默认并列展示 `MCP` 与 `Skills` 状态，而不是先做 tab 切换
- `设置` 页在 v1 中作为全局配置中心，而不是所有配置的杂物间

### MCP Status Card v1

`MCP` 状态卡片在 v1 中展示：

- 已配置的 MCP server 数量
- 已连接 / 失败 数量
- 每个 server 的状态
- 每个 server 提供了多少工具
- 一个“打开配置 / 管理”的入口

### Skills Status Card v1

`Skills` 状态卡片在 v1 中展示：

- 已发现的 skill 总数
- 来源分布：
  - builtin
  - user
  - project
  - plugin
- 最近 / 常用 skill 的简短列表
- 一个“管理 skills / 打开目录”的入口

### Default Main Page

`Xnova Studio v1` 打开后的默认主页面采用：

- 空白聊天页

说明：

- 中间主区保持类似 `Codex App` 的欢迎语 + 输入框形态
- 用户打开应用后可以立即开始干活
- 不采用项目启动向导页作为默认首页
- 保留少量、克制的建议动作入口（不做过多卡片和复杂引导）

### Suggested Actions on Blank Page

空白聊天页保留少量建议动作，v1 中只作为轻量快捷入口，不做复杂流程。

建议动作包括：

- 开始一个新项目
- 打开并继续一个已有项目
- 分析当前项目结构

v1 行为原则：

- 不做重型产品向导
- 不做过深的首页工作流
- 主要作用是帮助用户快速进入主聊天任务流

### Sidebar Main Blocks

左侧主体区分为两个核心块：

#### 项目

- 展示最近打开过的项目文件夹
- 每个项目下面承载该项目的会话管理
- 会话树与子代理会话挂在项目下面
- 默认展开到主会话层
- 子代理会话继续保持折叠

#### 聊天

- 作为即时工作会话入口
- 不强调项目层级归属
- 更偏“开始干活”的独立区域
- 采用轻量全局聊天列表 + 新建聊天入口

### Sidebar Block Behavior

左侧 `项目` 与 `聊天` 两个区块采用：

- 独立区块
- 各自支持折叠 / 展开
- 必要时各自独立滚动

### Main Context Near Input

输入框附近应长期显示：

- 当前项目 / workspace
- 当前分支
- 当前 agent profile
- 当前模型
- 上下文使用量
- 当前运行中的 `subagent` 数量

说明：

- 模型支持切换
- 按服务商分组展示该服务商下的模型
- 用户导入的所有兼容服务商都可以切换
- 当前模型/服务商切换后，新建会话默认跟随当前选择

显示顺序固定为：

1. 当前项目
2. 当前分支
3. 当前 Agent
4. 当前模型
5. 上下文使用量
6. 运行中的 `subagent` 数量

### SubAgent UX

`SubAgent` 首先是对话内事件与分支会话，而不是单独的后台管理面板。

表现方式：

- 主 Agent 在聊天流中派遣子代理
- 子代理创建与运行状态直接出现在当前对话里
- 左侧会话树支持父会话展开子代理会话
- 子代理会话可点击进入独立聊天视图

左侧层级规则：

- 父会话与子代理会话都支持显示
- 默认折叠在父会话下面

### Todo UX

- 主界面只显示执行中的 `todo` 摘要
- 完成后主视图摘要收起
- 任务明细仍保留在聊天记录或任务面板中可追溯

### Main Agent Selector UX

主 Agent 选择器采用：

- 默认简洁下拉框
- 支持进入完整的 `管理 agents` 面板

下拉框展示原则：

- 展示“角色感”，不展示“配置感”
- 分组使用用户语言：
  - `系统推荐`
  - `我的自定义`
- 每项默认只显示：
  - `name`
  - 简短副标题

下拉框默认不直接展示：

- `id`
- 完整 `when_to_use`
- `builtin / user` 这类实现细节词

### Manage Agents Panel v1

`管理 agents` 面板在 v1 先支持基础可编辑能力：

- 新建用户 agent
- 编辑用户 agent
- 删除用户 agent
- 新建 agent 支持两种路径：
  - 从模板创建
  - 从空白创建

### Agents Page v1

`Agents` 一级入口在 v1 中默认采用：

- Agent 列表页

说明：

- 先直接展示所有可用 agent
- 页面整体保持轻量、可编辑、易理解
- 列表按使用模式分组：
  - 主 Agent
  - 子代理
  - 通用
- 每个条目标识来源：
  - 内置
  - 自定义
- 每个条目默认展示：
  - `name`
  - `summary`
  - `mode`
  - 来源标签

### Agent Detail / Edit v1

点击某个 agent 后，v1 的详情/编辑页默认展示：

- 完整 frontmatter
- 完整 Markdown 正文

并支持直接编辑。

### Mode Switch

`XForge` 入口采用顶部显式模式切换：

- `Standard`
- `XForge`

模式切换含义：

- 共享同一项目、同一 workspace、同一上下文
- 改变的是执行策略和 workflow 行为
- 顶部模式切换是唯一的主切换入口
- 输入框附近的上下文信息条不再重复提供第二个 mode 切换入口，避免歧义
- 顶部默认只显示模式名称
- hover 或点击时显示一句简短模式说明

## Agent System Decisions

### Built-in Profiles

v1 默认内置 profile 先沿用 CLI 当前真实实现：

- `general`
- `explore`
- `plan`

### Custom Agent Support

必须支持：

- 用户自定义主 Agent
- 用户自定义 SubAgent / agent profile
- 桌面端直接创建、编辑、启用、切换
- 配置文件直接手写

### Agent File Format

采用：

- `Markdown` 主体
- `frontmatter` 元数据

原则：

- 核心字段少而刚好
- 预留 `extra` / `extensions` 扩展口

### Agent Frontmatter v1 Rules

v1 的 `agent frontmatter` 最小规则如下：

- `id`：必填；只允许英文小写、数字、连字符 `-`
- `name`：必填；允许中文、英文大小写与任意展示名
- `summary`：必填；用于 UI 的简短副标题展示
- `mode`：显式字段，或使用默认值
- `inherits`：可选；用于基于现有内置 agent 或用户 agent 复用
- `when_to_use`：必填
- `tool_policy`：必填，采用对象式结构
- `model_preference`：可选
- `extra`：可选；若存在必须是 object / table，v1 不校验内部 schema

推荐结构示意：

```toml
id = "explorer"
name = "Explorer"
summary = "只读探索"
mode = "all"
inherits = "explore"
when_to_use = "快速只读探索项目结构与关键模块"
model_preference = "fast"

[tool_policy]
mode = "include"
tools = ["read_file", "grep", "glob", "bash", "task_output"]

[extra]
```

### Agent Mode

Agent 增加显式使用模式字段：

```toml
mode = "primary" | "subagent" | "all"
```

语义：

- `primary`：只能作为主 Agent 使用
- `subagent`：只能作为子代理被派发
- `all`：两者都可以

默认规则：

- 未声明时默认 `all`

解析规则：

- `default_agent` 只能引用 `primary` 或 `all`
- 子代理派发候选池只允许 `subagent` 或 `all`
- 主 Agent 选择器只展示 `primary` / `all`
- SubAgent 配置器只展示 `subagent` / `all`

### Agent ID and Sources

- `default_agent` 等字段引用 `agent id`
- Agent 来源只有两类：
  - `builtin`
  - `user`

### Override Rule

覆盖规则只保留：

- `user > builtin`

说明：

- 不允许项目级定义新的 agent 文件
- 这样更干净，也更符合“自定义 agent 主要是跨多个项目复用”的定位
- v1 暂不做项目级 agent 可见性过滤
- 只要 agent 合法，默认都可见可选

## Config Decisions

### Unified Config Format

CLI 与桌面端配置统一使用 `TOML`。

不再以 `JSON` 作为主配置格式。

### Config Files

- 用户级：`~/.xnovacode/config.toml`
- 项目级：`.xnovacode/project.toml`

说明：

- 用户级全局默认策略必须存在
- 项目级配置用于局部覆盖与项目默认策略
- `config.toml` 与 `project.toml` 尽量保持同构 section 结构

### Provider Compatibility

`config.toml` 中的 `[providers]` 语义完全复用当前 CLI 设计：

- 支持 `openai` 兼容
- 支持 `anthropic` 兼容
- 允许用户自定义任意 provider 名称
- 现有 provider 相关字段语义保持不变，只把文件格式从 `JSON` 迁移到 `TOML`

### Provider Models Policy

v1 中每个 provider 的 `models` 继续由用户手动维护。

说明：

- 先完全复用 CLI 当前逻辑
- v1 暂不把“自动发现模型”作为默认能力
- 自动发现/写回/缓存可以作为后续增强方向

### Settings Page Priority

`设置` 页在 v1 中的优先顺序定为：

1. `Providers`
2. `Memory`
3. 其它基础偏好

### Providers Page v1

`Providers` 是设置页中最核心的一级块，v1 至少包含：

- 默认 provider
- 默认 model
- provider 列表

每个 provider 配置项至少包含：

- provider 名称
- `protocol`
- `api_key`
- `base_url`
- `models`
- `vision_models`

说明：

- API 供应商配置与模型 ID 列表配置统一归属于 `Providers`
- 模型列表在 v1 继续由用户手动维护
- `Providers` 页支持“测试连接”按钮
- `Providers` 页在 v1 中支持新增 / 编辑 / 删除自定义 provider

### Memory Config Compatibility

`config.toml` 中保留 `[memory]` section，并继续兼容 CLI 当前设计：

- `memory` 作为独立 section 存在
- 支持开关语义
- 支持 embedding 子配置
- 桌面端与 CLI 共享同一套 memory 配置语义

### Memory Module Positioning

`Xnova` 的 `Memory` 不应被弱化为隐式能力，也不应在 v1 被做成写死的大型知识库管理器。

v1 目标：

- 把 `Memory` 保持为显式一等模块
- 在桌面端做成“半管理页”
- 提供足够的配置、状态与维护能力

设计原则：

- 不写死当前实现
- 不把未来升级路径堵死
- 为后续更强的项目记忆、跨会话知识沉淀、检索策略升级预留空间

### Memory Default Policy

`Memory` 在 v1 中默认开启。

如果用户尝试关闭：

- 需要有显式提醒
- 建议保持开启

目的：

- 保持 `Xnova Studio` 的强 `project-aware` 能力
- 避免用户无意关闭后丢失关键体验

### Memory UX Principle

对普通用户，`Memory` 应保持低打扰，优先呈现收益而非机制。

用户应优先感知到：

- 更少重复解释
- 会话连续性更强
- 项目上下文更稳定

而不是被要求先理解：

- embedding
- 索引
- 向量检索
- 记忆库管理

因此：

- `Memory` 保持默认开启
- 平时低存在感
- 只在用户主动进入设置或发生配置/状态问题时，再展示更多机制细节

### Memory Page v1

`Memory` 页在 v1 中采用“半管理页”定位：

- 不是只有开关的轻配置页
- 也不是大型知识库管理后台

v1 至少包含：

- memory 开关状态
- embedding 配置
- 当前是否启用
- 当前索引 / 记忆状态
- 重建入口
- 项目记忆数量概览
- 全局记忆数量概览

设计原则：

- 比 `OpenCode / Aider / pi-coding-agent` 的隐式记忆方式更显式
- 但在 v1 中保持克制，不做过重的知识库运营面板

### Config Boundary

项目配置与 Agent 定义分离：

- `project.toml` 管项目默认运行策略
- `agent files` 管 Agent 本体定义与行为语义

### Config Precedence

配置优先级定为：

- 项目级 `.xnovacode/project.toml`
- 用户级 `~/.xnovacode/config.toml`
- 内置默认值

即：

- `project > user > builtin`

### Config Merge Strategy

v1 合并策略按字段类型区分：

- 标量字段：项目级覆盖用户级
- 对象字段：按 key 合并，冲突项以项目级为准
- 数组字段：v1 先采用“项目级整组覆盖”

说明：

- 这种策略吸收了 `OpenCode` 的分层 merge 思路
- 同时保持 v1 规则简单、可解释、可预测
- 对于 `enabled_features`、`allowed_modes` 这类项目策略字段，整组覆盖更符合直觉

### Minimal Project Config Fields

`project.toml` 的 v1 最小字段集合：

- `agent.default`
- `agent.max_parallel_subagents`
- `features.enabled`
- `modes.allowed`

其中：

```toml
[agent]
default = "general"
max_parallel_subagents = 5

[features]
enabled = ["subagents", "memory", "mcp", "skills", "plugins"]

[modes]
allowed = ["standard", "xforge"]
recommended = "xforge"
```

说明：

- `preferred_workflow` 暂不进入 v1 最小字段集合
- `XForge` 的项目可用性通过 `allowed_modes` 表达
- 可选增加：
- `modes.recommended` 只做推荐，不强制切换
- 当项目配置了 `modes.recommended` 时：
  - 首次进入该项目时弹一次轻量提示
  - 后续不重复打扰
- 一旦用户在该项目中手动切换 mode，则记住该项目最近一次 mode 选择
- 后续再次进入该项目时，优先恢复该项目最近一次用户 mode 选择
- 用户可以一键恢复到项目推荐 mode

### Default Agent Runtime Rule

当项目配置了 `agent.default` 时，v1 的运行时规则如下：

- 首次进入项目时，使用项目配置的 `default_agent`
- 之后如果用户在 UI 中手动切换主 Agent，则记住该项目最近一次用户选择
- 后续再次进入该项目时，优先恢复“该项目最近一次用户选择”
- 用户可以一键恢复到项目推荐/默认值

### Parallel SubAgent Rule

`agent.max_parallel_subagents` 在 v1 中定义为：

- 硬上限
- 系统不会超过这个数量

默认上限先定为：

- `5`

### Model/Provider Policy

- `project.toml` v1 不强制保留 `default_provider` / `default_model`
- Provider 和模型能力在全局配置中管理
- 用户当前选择的 provider/model 会影响新建会话的默认行为
- 桌面端模型切换应复用 CLI 现有 provider 兼容能力：
  - `openai` 兼容
  - `anthropic` 兼容
  - 用户自定义服务商

### Enabled Features v1 Enum

`features.enabled` 首批枚举定为：

- `subagents`
- `memory`
- `mcp`
- `skills`
- `plugins`

默认值建议定为：

```toml
[features]
enabled = ["subagents", "memory", "mcp", "skills", "plugins"]
```

运行原则：

- `enabled_features` 是强开关
- 不在列表中的能力，在该项目中不可用
- 但能力默认支持，不等于实例默认已经存在

例如：

- `mcp` 默认开启的是机制支持，不代表所有 MCP server 自动存在
- `skills` 默认开启的是机制支持，不代表当前一定发现到可用 skill
