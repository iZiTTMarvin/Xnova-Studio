# Studio 主链路修复 PRD

## 背景

当前 `studio/` 主壳已经具备 project-aware 的界面骨架，但主链路仍存在三处关键断点：

1. 进入项目会话后只能看到摘要卡片，看不到真实聊天消息流、工具过程与运行反馈。
2. `runtime-not-ready` 仅作为提示展示，未形成发送门禁，未绑定 Workspace 时仍可能误触发运行。
3. Studio 主界面缺少会话级模型选择，输入框旁边还会展示误导性的硬编码模型文案。

这三处问题直接阻断“在 CLI 中已可工作的核心体验，也能在 Studio 中正常使用”这一目标。

## 目标

- 打通 Studio 的真实会话聊天视图，让用户能在桌面端看到消息流、工具/运行反馈与会话恢复结果。
- 把 `runtime-not-ready` 从“提示”升级为“门禁”，未绑定 Workspace 时不能误提交到错误目录。
- 为 Studio 主工作页提供会话级 Provider / Model 选择，并让提交链路真正消费该选择。
- `XForge` 在当前阶段不做真实执行策略切换，但点击后必须给出明确的“暂未开放”反馈，避免伪开关。

## 非目标

- 当前阶段不实现真正的 `XForge` orchestration。
- 当前阶段不重做整套 Studio 视觉设计。
- 当前阶段不扩展外部 Agent Adapter 或新增平台化能力。

## 约束

- 必须沿 `studio -> preload/ipc -> runtime` 现有边界增量演进，不能让 renderer 直连底层单例。
- 不能只做 UI 占位；用户的操作必须能进入真实运行时主链路。
- 需要与现有 CLI 的 provider 兼容与 runtime submit 行为保持一致。

## 交付范围

### 1. 会话聊天视图

- `activeSession` 页面展示真实消息流，而不是仅展示摘要卡片。
- 支持展示用户消息、助手消息、系统/警告信息，以及本轮运行反馈。
- 保持与项目树/子代理树联动，不破坏当前 project-aware 信息架构。

### 2. runtime 门禁

- 未绑定 Workspace 时，Composer 发送按钮禁用。
- `submitPrompt()` 在前端做显式拦截并给出用户可见反馈。
- main/runtime 服务不再回退到 `process.cwd()` 静默执行桌面主链路请求。

### 3. 会话级模型选择

- 主工作页输入区提供 Provider + Model 选择能力。
- 当前选择进入 runtime submit 请求，而不是只修改“默认模型”设置。
- 上下文条与输入区文案不再展示硬编码假模型。

### 4. XForge 暂未开放反馈

- 顶部模式切换点击 `XForge` 时展示明确提示，说明当前阶段尚未开放。
- 不允许界面看起来像已经切换成功。

## 验收标准

- 用户进入已有项目会话时，可以直接看到真实聊天消息，而非只有统计摘要。
- 未绑定 Workspace 时，用户无法提交项目级提示词，并能看到明确原因。
- 用户在主界面切换 Provider / Model 后，新提交立即按该选择执行。
- 顶部点击 `XForge` 时出现“暂未开放”的明确提示，不改变实际执行模式。
- `studio` 测试、`studio` 类型检查、`cli` 类型检查通过。
